Section 1

The Problem We're Solving

Imagine four people building a puzzle — but they're all sitting at the same table, reaching over each other, accidentally moving each other's pieces. Chaos.

Software teams face the exact same problem. If everyone edits the same code at the same time, work gets overwritten, bugs appear from nowhere, and nobody knows what changed or when.

This guide explains exactly how our team avoids that — using three tools you will hear about throughout this document:

 ✦  GitHub — where all our code lives and is saved. Think of it as our shared filing cabinet in the cloud.
 ✦  Railway — the platform that takes our code and turns it into a live, working website or dashboard.
 ✦  Claude (Hermes) — the AI assistant in the terminal that runs all the technical commands on your behalf. When this guide says "tell Hermes to do X", that means you type a plain English instruction and it handles everything behind the scenes.

The goal: Every person works independently in their own space, all progress is always visible, and good work can be shared between projects without breaking anything — all without needing to know a single line of code.

Section 2

The Sandbox — Your Own Private Workspace on Railway

In our world, a sandbox is a Railway environment. Let's explain what that means from scratch.

When we set up a project on Railway, we don't just get one version of it — we can have multiple environments running at the same time, each with its own live URL, each completely separate from the others. For example:

 ✦  production — the live dashboard your clients or stakeholders actually see
 ✦  staging — a private copy where we test changes before they go live
 ✦  preview — a temporary environment to run someone else's work before deciding whether to bring it in

Each environment is its own sandbox. You can break the preview environment completely and production never flinches. Nothing crosses between them unless you deliberately move it.

Think of each project like a separate LEGO set stored in its own box. Jarryd's box only contains Jarryd's pieces. Employee Y in America is working in a completely separate box on Railway. A manager can look into any box at any time — but just by looking, they can't accidentally move any pieces.

In our tools: The sandbox is your Railway environment. GitHub holds the code. Railway takes that code and makes it live. Each person has their own GitHub repository and their own Railway environment — they are completely independent. Hermes (Claude) is the one who creates, manages, and deploys these environments when you ask.

Each person has their own Railway environment and GitHub repository — completely isolated. No accidental interference.

Section 3

The Save System — How GitHub Remembers Everything

Every time a meaningful piece of work is finished — even something small — it gets saved to GitHub with a label. In GitHub's language this is called a commit. You do not need to remember that word — just think of it as a named save point.

Each save on GitHub automatically records what changed, who made it, and exactly when. This means we can always rewind to any earlier version if something goes wrong. Nothing is ever permanently broken.

When Hermes (Claude) finishes a piece of work with you — say, adding a new section to a dashboard — the final step is always to save it to GitHub and then tell Railway to redeploy. That three-step process (save to GitHub → Railway picks it up → live site updates) is the heartbeat of everything we do.

Analogy: Imagine writing an essay in Google Docs. Every few paragraphs you name and save a version — "Draft 1", "After feedback", "Final". GitHub is exactly that, but for code. And Railway is the system that publishes whichever version you point it at — instantly making it live for anyone with the link.
In practice: You never touch GitHub or Railway directly. You tell Hermes what you want in plain English — "save this and deploy it" — and it runs the commands in the terminal behind the scenes. GitHub and Railway are the engines. Hermes is the driver. You are in the passenger seat giving directions.

Every meaningful change is saved as a labelled snapshot — nothing is ever truly lost.

Section 4

Sharing a Piece — Borrowing What Works

Here's where the real power comes in. Let's say Alex builds a really clean way of displaying a financial summary table. Sam is working on a different project and thinks: "That would be perfect for mine."

Sam can copy that specific piece into her own project — without touching Alex's project, without breaking anything, and without removing it from Alex's version. Both teams keep it.

It's an ongoing connection — if Alex improves that piece later, Sam can choose to pull in those improvements too. The two projects stay in sync where it matters.

Analogy: Think of it like a shared recipe book. Alex writes a great sauce recipe. Sam copies it into her own cookbook. Both have it. Alex can keep improving hers, and Sam can decide when to update her copy.

A feature can move from one project to another without disrupting either team.

Real-World Scenario

You Work Days. Employee Y Works Nights. No Problem.

Here's a practical example of how this all works in the real world.

You spend your day building out a financial report. Employee Y, based in America, jumps on in the evening and makes a series of their own changes to their version of the project. By the time you're back at your desk the next morning, they've done a bunch of work.

Now you have two options and you need to decide: "Which of their changes do I actually want? And which ones do I want to leave out?"

The setup is called a Fork. You have your own independent copy of the project. Their copy is labelled the "original" (called the upstream). You can see every change they've ever made — but nothing they do automatically lands in yours. You decide what comes in and when.

Both timelines run independently. You pick which of their changes you want — nothing is automatic.

In practical terms this means:

 ✦  You can see every branch and change Employee Y has made
 ✦  You preview their changes before deciding
 ✦  You bring in only what you want — one piece at a time if needed
 ✦  Your project stays unaffected until you decide otherwise
 ✦  If you later want their change 4, you can pull it in at any time

The golden rule: Nothing crosses from their project into yours automatically. You are always in control of what enters your project and when.

Employee Y makes a change you're unsure about Preview it first in a test environment You see it live before it touches your main project
Employee Y makes a change you don't want Do nothing It stays in their project, never affects yours

Before Anything Else

Getting Access to Each Other's GitHub

Before you can see Employee Y's branches, borrow their code, or share yours with them — you need to be able to see each other's GitHub repositories. This section explains how that works from scratch.

Two Types of Repositories

Every project on GitHub is stored in a repository — think of it as a folder in the cloud that holds all the code, all the save points, and all the branches for one project. Every repository is either:

TypeWhat it meansWho can see it
Public Anyone on the internet can view it No invitation needed — just visit the link
Private Hidden — only people explicitly invited can see it Must be added as a collaborator by the owner
In our team: Most working project repositories are private. This means you and Employee Y need to actively invite each other before either of you can see the other's code. This is a one-time setup — once done, it stays in place.

How to Invite Someone to Your Repository

The owner of a repository controls who gets access. Here is the step-by-step — no technical knowledge required:

What to share: All you need is the other person's GitHub username. That's their handle on GitHub — for example jarrydgreen-cleverprofits. Ask Employee Y for theirs, share yours with them, and Hermes can handle the rest if needed.

Two Ways to Connect to Someone's Repository

Once you can see someone's repository, there are two ways to connect to it depending on your goal:

MethodPlain EnglishWhen to Use It
Fork Create your own independent copy of their entire repository — with a permanent live link back to theirs When you want to build your own version of someone's project and regularly pick up their improvements over time
Add as Upstream Tell your existing repository "keep an eye on this other person's repository" When you already have your own repo and just want to be able to pull specific pieces from theirs
In our scenario: Jarryd works on their own dashboard project during the day. Employee Y in America works on their version in the evenings. Jarryd's repository has Employee Y's repository added as an upstream — meaning Jarryd can see every branch and every save Employee Y has made, at any time, and choose exactly what to bring in. Employee Y's project is unaffected either way.

How to Get the Information — Fetching

Once access is set up, getting Employee Y's latest work onto your machine is a single instruction to Hermes: "Fetch Employee Y's latest branches."

What happens behind the scenes:

 ✦  GitHub downloads a copy of all their branches silently to your machine
 ✦  Nothing in your project changes — you are just downloading so you can look
 ✦  Hermes then lists what branches they have, with a plain English summary of each
 ✦  You decide which ones interest you — then move to the preview or cherry-pick steps

Remember: You never do any of this manually. You tell Hermes in plain English — "Fetch Employee Y's latest" or "show me their branches" — and it handles everything. GitHub and Railway are working in the background. You just make the decisions.

How To

Getting Employee Y's Work Into Yours — Step by Step

Employee Y has been building on their project. He's created branches on GitHub. Here's exactly how you get what you want from their project into yours — safely, selectively, and without touching anything you didn't choose to touch.

Remember: Railway deploys from your GitHub. Their code never goes directly to your live site — it always passes through your GitHub first. You are always the gatekeeper.

What Is Cherry-Picking?

Imagine Employee Y made 10 changes last night. They're all bundled together in their branch. You look through them and think:

"I love changes 3 and 7. Changes 1, 2, 4, 5, 6, 8, 9, and 10 I'm not ready for yet."

Cherry-picking lets you take exactly changes 3 and 7 — nothing else. They land in your project cleanly, as if you had written them yourself. The other 8 changes stay in their project untouched.

Analogy: Think of Employee Y's branch like a fruit bowl with 10 pieces of fruit. You don't have to take the whole bowl. You pick the two cherries you want, leave the rest in their bowl, and put your two cherries into your own bowl. Their bowl is still full. Yours now has exactly what you wanted.

Cherry-picking = surgical precision. Take exactly what you want, nothing more.

Why Cherry-Picking Matters

Without cherry-picking, your only options would be: take everything or take nothing. That's a problem because:

 ✦  Employee Y might bundle a feature you love with a change that breaks your layout
 ✦  You might not be ready for all their changes yet — but you need one specific piece now
 ✦  You might want to test one piece in isolation before committing to the rest

Cherry-picking solves all three problems. It is arguably the most important tool in a collaborative coding workflow.

What Needs to Be in Place for Cherry-Picking to Work

Cherry-picking only works cleanly when certain conditions are met. Think of it like surgery — it needs a clean operating environment.

Requirement Plain English What Happens If Missing
Clean, small saves (commits) Each save should do one thing only — not ten things at once You can't pick the one piece you want without dragging in everything else
Descriptive save labels Each save should have a clear name — "added profit margin column" not "update" You won't know which save contains what you're looking for
Access to their repository You need to be able to see their GitHub project (public or invited) You can't fetch their branches if you can't see their repo
No conflicting changes You and Employee Y shouldn't have both edited the exact same line of code A "conflict" occurs — needs manual resolution before the cherry-pick lands
The golden rule for Employee Y: Ask them to make small, focused saves with clear labels. One feature per save. This makes cherry-picking trivial — you can see exactly what each save does and take it or leave it in seconds.

Merge or Cherry-Pick — How to Decide

Cherry-picking and merging aren't two different approaches — they're just two points on the same scale. Here's how to decide which one to use:

SituationWhat You DoWhat It Means
You like everything in Employee Y's branch Merge the whole branch The entire branch lands in your project at once — fast and complete
You like some things in Employee Y's branch Cherry-pick the specific commits you want Only the pieces you choose come in — everything else stays in their project
You like nothing in Employee Y's branch Do nothing Your project stays completely untouched
How to action it: Just tell Hermes in plain English — "merge Employee Y's Income-Forecast branch into mine" or "I only want their profit margin change." Hermes handles the rest.

What Is a Conflict — and Should You Worry?

A conflict happens when both you and Employee Y have edited the exact same line of code in different ways. Git doesn't know whose version to keep, so it flags it and asks you to decide.

Analogy: You and Employee Y both rewrote the same sentence in a shared document at the same time — differently. Now someone has to decide which version stays. That's a conflict. It sounds scary but it's normal — and Hermes handles it for you.

In practice, conflicts are rare if you and Employee Y are working on different parts of the project. The more separated your work areas, the smoother cherry-picking becomes.

Visibility — Who Can See What?

A common concern: "If everyone is working separately, how do I know what's going on?"

The answer is: full visibility, zero interference.

A team lead or manager can view every project from a single dashboard. You can see the live version of anyone's work at any time, a complete history of every change ever made, and leave comments and feedback — without touching any code.

Analogy: It's like being able to watch any employee's screen in real time — without moving their mouse. You see everything, you change nothing.
